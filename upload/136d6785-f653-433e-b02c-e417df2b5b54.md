## 1. Project Overview

**Goal:** Build a web service that converts BVH (motion capture) files to FBX format for import into Unreal Engine 5, with **root motion preservation** and **embedded skeleton**.

**Critical User Requirements:**
1. The skeleton must be **EMBEDDED** in the output FBX — UE5 must show the import dialog asking to create a new skeleton or use an existing one
2. **Root motion** must be preserved — the Root bone carries translation, skeleton root (Hips/pelvis) carries rotation only
3. Two target skeleton variants: **X Bot (Mixamo, 65 bones)** and **UE5 Quinn Mannequin (88 bones)**
4. The user is Russian-speaking — all UI text and responses should be in Russian when interacting

## 4. The FBX Export Problem — Detailed History

### 4.1 V1: Custom FBX ASCII Writer (`bvh_to_fbx.py`) — BROKEN
- Produced FBX ASCII 7.5.0 format
- **Result:** UE5 shows "Cannot open FBX file" / "Nothing to import" for Quinn variant
- X Bot variant: Opens import dialog but "Import failed"
- 3dviewer.net shows "Unknown format"
- **Root cause:** Custom FBX ASCII writer produces malformed files — incorrect node structure, missing required records, wrong connection format

### 4.2 V2: Custom FBX Binary Writer (`bvh_to_fbx_v2.py`) — BROKEN
- Produced FBX 7500 binary format
- Reads reference skeleton binary, retargets BVH animation onto it
- Has `FBXBinaryReader` + `FBXBinaryWriter` classes
- **Result:** Still produces invalid/malformed FBX — UE5 cannot import
- **Root cause:** Custom binary writer has bugs in record serialization, connection format, animation curve encoding

### 4.3 V3: Blender Python API (`bvh_to_fbx_v3.py`) — CURRENT, NEEDS VERIFICATION
- Uses `bpy.ops.import_scene.fbx` to import reference skeleton
- Uses `bpy.ops.import_anim.bvh` to import BVH animation
- Retargets animation from BVH armature to reference armature via world-space rotation matching
- Adds "Root" wrapper bone for UE5 root motion extraction
- Uses `bpy.ops.export_scene.fbx` for valid binary FBX output
- **Blender's internal re-import validation passes**
- **But user reports UE5 still cannot import** — this is the CURRENT BLOCKER

### 4.4 Possible reasons for V3 failure in UE5:
1. **Blender FBX export settings may not be optimal for UE5** — wrong axis conversion, bone axis settings, or armature node type
2. **Root bone structure may confuse UE5** — the Root wrapper + original root may not be recognized properly
3. **Animation bake settings** — `bake_anim_simplify_factor`, `bake_anim_step`, `bake_anim_use_all_bones` may need tuning
4. **FBX version compatibility** — Blender exports FBX 7400, UE5 may need specific settings
5. **Missing Pose data** — the rest pose may not be stored correctly
6. **Scale/transform issues** — the reference skeleton has 0.01 scale factor that may cause issues in UE5

## 7. Root Motion Implementation

### Current Approach (V3):
1. **Root wrapper bone ("Root")** is added ABOVE all current root-level bones
2. Root bone has ONLY translation keyframes (the root motion displacement)
3. Skeleton root bone (Hips/pelvis) has ONLY rotation keyframes
4. Root motion displacement = (BVH root world position at frame N) - (BVH root world position at rest)
5. Displacement is converted from BVH coordinate space to reference skeleton local space
6. Scale conversion: `motion_scale = ref_armature_scale / bvh_armature_scale`

### UE5 Root Motion Extraction:
- In UE5, the animation sequence must have "Enable Root Motion" checked
- The "Root" bone's translation animation is extracted as root motion
- The character moves according to the Root bone's translation delta between frames
- Without Root bone, the character walks in place

## 10. Recommended Next Steps (Priority Order)

### Step 1: Diagnose UE5 Import Failure 🔴
**The most important step.** The V3 converter using Blender produces FBX that Blender can re-import but UE5 cannot. Need to:
1. Compare the output FBX with a known-working UE5 animation FBX (like `TP_Echo_Walk_Fwd.FBX`)
2. Use `python3.13` to analyze both files with `bpy` — what's different?
3. Check specific Blender export settings that matter for UE5:
   - `armature_nodetype` — should be `'ROOT'` (currently set)
   - `primary_bone_axis` / `secondary_bone_axis` — may need adjustment
   - `apply_scale_options` — may need `'FBX_SCALE_ALL'` or `'FBX_SCALE_NONE'`
   - `use_space_transform` — may need to be `False` for UE5
   - `object_types` — currently `{'ARMATURE'}`, should include mesh if needed
4. Try exporting without the Root wrapper bone first — does the base skeleton+animation import?
5. If it does, the issue is with the Root bone structure

### Step 2: Fix Blender FBX Export Settings
Based on diagnosis, adjust `bvh_to_fbx_v3.py` export settings. Key areas:
- Axis conversion (currently `-Z` forward, `Z` up for UE5)
- Bone axis configuration
- Scale handling (the 0.01 scale factor in reference skeletons)
- Animation bake settings

### Step 3: Alternative Approach — Use fbxsdk
If Blender-based export continues to fail, consider switching to the official Autodesk FBX SDK:
- Check if `fbxsdk` Python package is available or can be installed
- The FBX SDK produces the most compatible FBX output for UE5
- Would need to rewrite the converter to use `fbxsdk` instead of `bpy`
- This is the nuclear option but most reliable

### Step 4: Test with UE5
After any fix, the output must be tested in actual UE5:
1. Drag FBX into Content Browser
2. UE5 should show the import dialog (skeleton creation/selection)
3. Select "Animation" import type
4. Verify animation plays correctly
5. Verify root motion works (character moves forward, not in place)

### Step 5: Address Server Stability
- Consider using a Blender headless server process instead of spawning new processes
- Or implement proper cleanup between conversions
- Add timeout and error recovery in the API route

## 11. Key Code Sections to Understand

### 11.1 V3 Converter Flow (`bvh_to_fbx_v3.py`)
```
1. Clean Blender scene (read_factory_settings, remove all objects)
2. Import reference skeleton FBX → ref_armature
3. Import BVH animation → bvh_armature (with action)
4. Build bone mapping (MIXAMO_MAP or UE5_QUINN_MAP + fuzzy matching)
5. Find BVH root animation bone (typically "Hips")
6. Compute root motion scale (ref_armature_scale / bvh_armature_scale)
7. Add "Root" wrapper bone above ALL root-level bones
8. Create new animation action on ref_armature
9. For each frame:
   a. Compute root motion displacement from BVH
   b. Set Root wrapper location = displacement
   c. Set ref root bone rotation from BVH
   d. For all other mapped bones: set rotation from BVH (world-space to local-space)
10. Clean up (remove BVH armature, meshes)
11. Export FBX with Blender's FBX exporter
12. Validate by re-importing in Blender
```

### 11.2 API Convert Route Flow (`src/app/api/convert/route.ts`)
```
1. Receive FormData with BVH file + settings
2. Save BVH to temp file (UUID-based naming)
3. Determine skeleton file path (built-in or uploaded custom)
4. Execute: python3.13 bvh_to_fbx_v3.py <bvh> <skeleton> <output> --scale --axis --skeleton
5. Read output FBX file
6. Read stats JSON file
7. Clean up temp files
8. Return FBX binary as download with metadata headers
```

### 11.3 Root Bone Addition Logic (`add_root_bone()`)
```
1. Find ALL current root-level bones (no parent)
2. Create new "Root" edit bone at primary root's position
3. Make ALL root-level bones children of Root
4. Set Root's rotation mode to ZYX
5. Set Root's default location to origin
```
